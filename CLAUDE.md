# CLAUDE.md — Stage-1-Issues-Study

A static HTML site (chamber.html, bank.html, drafting.html, explainer.html, resources.html, using.html, index.html) plus one Netlify Function (`netlify/functions/chat.js`) that proxies to Claude Haiku 4.5. It is a Socratic AI tutor for Year 11 SACE Philosophy students working on their Issues Study assessment. Peter is the teacher; students are ~14-30 per class.

The sister Stage 2 lab (Year 12) lives in a separate repo and uses a separate API key.

## Architecture in one screen

```
bank.html (curated questions)
  → student pins a question
  → chamber.html opens a Socratic dialogue
  → chat.html UI POSTs to /.netlify/functions/chat with { messages, pack, workingQuestion }
  → chat.js builds system prompt + readings + history, calls Anthropic, returns reply
  → drafting.html / explainer.html / resources.html for follow-on writing tasks
```

Working question persists across the session via localStorage and is passed in every request.

`chat.js` constants (do not change without reason):
- `MODEL = 'claude-haiku-4-5'` — hardcoded. Haiku resists over-explaining, which is the right Socratic discipline. Smarter models lecture.
- `MAX_OUTPUT_TOKENS = 600`
- `HISTORY_TURN_LIMIT = 6` (last 12 messages sent)
- `RATE_LIMIT_MAX = 30` per IP per 5 min — soft, in-memory per Lambda container

Reading packs (`data/packs/*.txt`, plain UTF-8) range 20K-68K tokens. They're bundled into the function via `netlify.toml`'s `[functions] included_files = ["data/packs/*.txt"]`. Without that line, `fs.readFileSync` fails silently and packs degrade to topic-context strings.

## Cost defences (encoded — do not weaken)

1. Haiku 4.5 only.
2. `cache_control: { type: 'ephemeral', ttl: '1h' }` on the readings block (system message). 1h TTL costs 2× to write but lasts 12× longer than the 5-min default.
3. Multi-turn caching: most recent assistant turn also gets a 1h `cache_control` marker, only when readings are present (otherwise prefix is below Haiku's 2048-token cache minimum).
4. Cache marker is skipped entirely when no pack is selected — the bare system prompt (~400 tokens) would silently no-op the marker.
5. History truncation to 6 turns. Output cap 600 tokens.
6. Per-IP sliding-window rate limit (soft; sufficient at class scale).
7. Per-request structured logging in Netlify function logs: `{ event, ip, pack, input_tokens, output_tokens, cache_creation_input_tokens, cache_read_input_tokens }`. Use this to verify cache hits.

### The May 1 2026 incident (why TTL matters)

Anthropic console for May 1 (key `key-for-claude`, ~14 students hitting at once during a lesson) showed a 1:1 cache_write:cache_read ratio on Haiku. Every cached chunk was rewritten as soon as it was used once. Root cause: default 5-min TTL + student work pattern (read answer → think 6+ min → ask follow-up) → cache always expired by the next turn. Day cost ~$5.60 just on the 5-min ephemeral writes of the readings packs. Stage 2 sister-lab key on May 6 had a 1:2 ratio at smaller volumes — the comparison baseline.

Fix shipped in commit `299f9eb` on `claude/investigate-api-costs-Jid0n`: 1h TTL + multi-turn caching + skip marker when prefix < min. Expected ~40-50% saving on busy days. **Verify by pulling a new console CSV ~7-10 days post-merge: cache_write_5m should drop, cache_write_1h should appear, cache_read should rise to 1:5 or better.**

## Pedagogical guardrails (encoded — do not soften)

Source of truth: `SOCRATIC_SYSTEM_PROMPT` in `chat.js:34-61`. Core rules:

- Never write the question for the student. Never write the essay. Never summarise readings unprompted.
- One focused question per turn, then stop. 2-4 sentences usually.
- Convert vague topics into questions. Convert questions into "what positions exist?". Convert single positions into "what's the strongest objection?".
- Point to the reading rather than summarising: *"Have a look at the Murdoch passage on attention; what do you think she'd say to that?"*
- Use philosophical terminology accurately (bad faith, eudaimonia, intentional fallacy, qualia, deontological).
- No generic praise ("great question!", "well done!"). Engage at the level of thinking: *"That's a sharper formulation."*
- If asked to write the essay, refuse warmly: *"That would do the thinking the assessment is asking *you* to do."*

## Anti-patterns Peter has explicitly rejected

- A "summarise this reading" button.
- Praise vocabulary.
- Complete philosophical questions generated by AI.
- Multi-paragraph AI replies.
- Models smarter than Haiku (they lecture).
- Sending images/PDFs/scans when text is available. The `readings_source/` folder contains source PDFs but they NEVER reach the API — only pre-extracted text in `data/packs/*.txt` does.
- Caching markers on prefixes below model minimum (no-op + wasted slot).

## Netlify (Pro plan)

Five jobs Netlify does for this site:
1. Hosts static HTML on its CDN.
2. Routes `/api/chat` → `/.netlify/functions/chat` via `netlify.toml` redirect.
3. Bundles `chat.js` + Anthropic SDK + the included `data/packs/*.txt` with esbuild.
4. Runs the bundle on AWS Lambda when invoked.
5. Stores `ANTHROPIC_API_KEY` env var, injected at runtime.

### Pro-plan facts (verified 2026-05-10)

- **Function timeout**: 60s default and max for synchronous functions (cannot be customised, even on Pro). Background functions allow 15min via async invocation. *(Earlier handoff memo mentioning "26s, request via support" was outdated — current docs are clear.)*
- **Site password protection**: yes on Pro. ONE password, anyone with it gets in. Dashboard path: *Project configuration → Access & security → Visitor access → Password Protection → Configure → Basic password protection*. Choose scope: All deploys (production + non-production) or Non-production deploys only. Source: <https://docs.netlify.com/manage/security/secure-access-to-sites/password-protection/>
- **Password protection covers `/.netlify/functions/*`**: yes — the docs are silent but multiple support-forum threads confirm it gates functions too (people complain it breaks webhooks). For a copyright-defence + cost-defence use case with no external integrations, this is exactly what you want. ONE password = both HTML and chat function gated.
- **Netlify Identity**: officially deprecated 2026 (still available, no further development beyond security fixes). Successor: Auth0 extension for Netlify. For "one shared password for ~30 students", basic password protection is simpler than Auth0.
- **Log drains**: Enterprise only. Pro must rely on the 7-day function-log retention in the Netlify UI. (Earlier handoff said "available on Pro" — that was wrong.)
- **Deploy contexts**: production / branch-deploy / deploy-preview each get distinct URLs and can have distinct env vars (e.g., a separate test `ANTHROPIC_API_KEY`). Useful for testing API changes without affecting live class traffic. Set in *Project configuration → Environment variables → Add variable → Specific scope*.

### Cold-start gotcha

`packCache` and `rateBuckets` Maps in `chat.js` are per-Lambda-container. Two parallel students may land on two containers, each with empty Maps. `packCache`: small perf hit on cold start. `rateBuckets`: rate limit is SOFT — banned IP on container A is not banned on container B. Fine at class scale.

### Deploy mechanisms

The site auto-deploys from `main` via Netlify's GitHub integration. Other paths code can reach Netlify:
- `netlify deploy` CLI (uses a personal access token stored at `~/.netlify/config.json`).
- Netlify API (`Authorization: Bearer <token>`).
- Netlify Drop (drag-drop ZIP, manual UI action).

A Claude Code session with a Netlify access token in env or in CLI config could deploy out-of-band without git. **This is how the orphaned `feedback` function was likely deployed** — see next section.
